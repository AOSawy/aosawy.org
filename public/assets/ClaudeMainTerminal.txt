  - What: Astro — purpose-built for content-driven sites. Ships zero JS by default, blazing fast. Content collections from Markdown/YAML files.
  - Pros: Perfect Cloudflare Pages integration, content collections are exactly the hybrid model you want (publications.yaml, news posts as .md files), great
  component ecosystem, built-in image optimization
  - Cons: Smaller community than Next.js, less ecosystem if you ever need dynamic features
  - Best for: Exactly this use case — content-heavy, mostly static, data-driven

  Approach B: Next.js (Static Export)

  - What: Next.js with static export (output: 'export'). React-based, huge ecosystem.
  - Pros: Massive community, easy to find components/templates, you could add dynamic features later (API routes, auth), most transferable skillset
  - Cons: Heavier than needed for a static site, more boilerplate, static export mode disables some best features (ISR, middleware). Overkill for a portfolio site.
  - Best for: If you anticipate needing server-side features later

  Approach C: Hugo (Ultra-Minimal Static Generator)

  - What: Hugo — Go-based, extremely fast builds, Markdown-first.
  - Pros: Fastest build times of any SSG, dead simple content model, mature academic themes exist, zero JS dependency
  - Cons: Go templating is awkward, harder to customize design, limited component model, less flexible than Astro for custom layouts
  - Best for: If you wanted minimal tooling and maximum simplicity
